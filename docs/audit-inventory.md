## 4. 自有角色种子与已有库补权

### 4.1 当前机制

1. `BUILTIN_SELF_ROLE_PERMISSIONS` 在运行时代码中由 `ALL_SELF_PERMISSIONS`、`ADMIN_SELF_PERMISSIONS`、`MEMBER_SELF_PERMISSIONS` 组装：owner=全部，admin=除 SSO/logs 外的全部，member=固定业务权限集。见 `packages/server/src/iam/self/rbac/permissions.ts:91-151`。
2. 角色不是在注册时 seed。四库 `AddFlowOpsIamEntities` 迁移预插入 owner/admin/member，初始 permissions 为 `[]`：postgres `packages/server/src/database/migrations/postgres/1778000000000-AddFlowOpsIamEntities.ts:99-106`，mysql `:96-101`，mariadb `:96-101`，sqlite `:98-105`。
3. 后续 `BackfillFlowOpsRolePermissions1778000100000` 从独立常量写入完整 permissions；helper 定义见 `packages/server/src/database/migrations/flowOpsIamRolePermissions.ts:1-145`，四库迁移循环见 postgres `packages/server/src/database/migrations/postgres/1778000100000-BackfillFlowOpsRolePermissions.ts:4-13`、mysql `packages/server/src/database/migrations/mysql/1778000100000-BackfillFlowOpsRolePermissions.ts:4-13`、mariadb `packages/server/src/database/migrations/mariadb/1778000100000-BackfillFlowOpsRolePermissions.ts:4-13`、sqlite `packages/server/src/database/migrations/sqlite/1778000100000-BackfillFlowOpsRolePermissions.ts:4-13`。
4. 注册首管只调用 `getRole(manager, 'owner')`；角色缺失会报 500，不会现场 seed。见 `packages/server/src/iam/self/auth/service.ts:173-202`、`:439-442`。
5. 这些迁移同时注册在 full 与 ship 集；例如 postgres `index.ts:54-55,114-115` 和 `index.ship.ts:54-55,114-115`，其余三库结构相同。

### 4.2 选定方案：版本化迁移，不做启动对账

选择 **四库幂等迁移**，不选每次启动对账，理由如下：

-   权限变更是 schema/data 版本的一部分，迁移有执行记录、可测试、可回滚，不在每次启动时隐式改数据。
-   A1 本就要求更新四库 full/ship 迁移集；A3 在同一机制中追加权限 migration，部署行为一致。
-   启动对账会在每次起服覆盖管理员对内建角色的合法调整，且失败会污染启动职责。

具体方案：

-   在 `SELF_PERMISSION_GROUPS` 新增 `auditLogs: definePermissions(['auditLogs:view'])`。owner 自动继承；admin 也自动继承；member 不添加。
-   同步更新 `flowOpsIamRolePermissions.ts` 的历史内建权限常量，保证全新库 seed 后已含新权限。
-   新建四库同时间戳迁移，仅查询 `isBuiltin=true AND name IN ('owner','admin')` 的行，在 TypeScript 中解析 permissions JSON，缺少时 append `auditLogs:view` 后更新；已有则不变。不要像旧 backfill 那样整段覆盖 permissions。
-   `down()` 只从 owner/admin 移除 `auditLogs:view`，保留其他权限。迁移同时注册各 driver 的 `index.ts` 与 `index.ship.ts`。
-   测试两类数据库状态：全新库 owner/admin 有权限、member 无；模拟旧库保留原权限并只补一个新 key，重复执行结果不变。

注：服务端 `checkPermission()` 允许 organization admin bypass，但普通 admin 和 UI 仍依赖 permissions 列表，因此仍必须对 owner/admin 都做数据补权：`packages/server/src/iam/self/middleware.ts:41-44`、`packages/ui/src/routes/RequireAuth.jsx:56-75`。

## 5. login-activity 统一方案

选定 **`flowops_audit_log` 为唯一运行时真源，废弃旧实体的运行时读写，但暂留旧物理表一个兼容周期**，不保留两套并行记录。

| 阶段  | 处理                                                                                                                                                                                                                                                              |
| ----- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| A1    | 创建 `flowops_audit_log`；迁移已有 `flowops_login_activity` 数据到新表。code `0` -> `auth.login/success`，`1` -> `auth.logout/success`，负数 -> `auth.loginFailed/failure`，原 code/message 放 metadata；能查到 user 时回填 actorEmail。旧表暂不 drop，便于回滚。 |
| A2    | 所有新 auth 事件只写 `flowops_audit_log`；停止调用/移除 `recordLoginActivity()`，因此不会重复记录。                                                                                                                                                               |
| A3    | `/audit/login-activity` 如需阶段兼容，也必须改为从新审计表按 `action LIKE 'auth.%'` 查询；新主接口为 `GET /api/v1/audit` 和 `/audit/export`。旧表不再读。                                                                                                         |
| A4    | 新审计页接新接口；`/login-activity` 可重定向到带 auth filter 的 `/audit`，随后移除旧 UI/API。                                                                                                                                                                     |
| A5 后 | 确认升级与回滚窗口结束后，再以独立清理迁移 drop `flowops_login_activity`。本计划内可先移除 TypeORM entity/runtime alias，但不在没有人工确认时物理丢弃历史表。                                                                                                     |

这样既只有一条新写入链路，又保留历史登录记录和发布回退能力。A0 需要人工确认的唯一保守点是“旧表延迟一个发布周期再 drop”；若要求本计划内物理删除，应把 drop 放在 A5 最终迁移并先验证历史数据迁移计数。

## 6. 功能门禁决策

建议 **新增 `feat:audit`，不复用 `feat:login-activity`**：完整审计包含角色、工作区、成员、组织用户和 auth 事件，继续沿用“登录活动”命名会把产品边界锁死。

| tier       | `feat:audit` | 理由                                                                                                                                               |
| ---------- | ------------ | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| free       | 关闭         | 保持核心免费能力，不开放管理审计。                                                                                                                 |
| pro        | 关闭         | 当前 pro 只有 datasets/evaluations/evaluators/logs，没有组织管理套件。`packages/server/src/services/entitlement/catalog.ts:53-55`                  |
| team       | 开启         | 当前 users/workspaces/roles/login-activity 已从 team 开始开放，完整审计与团队管理同档。`packages/server/src/services/entitlement/catalog.ts:56-65` |
| enterprise | 开启         | enterprise 包含全部 team IAM 管理能力。`packages/server/src/services/entitlement/catalog.ts:66-76`                                                 |

A3 在 `IAM_FEATURES_BY_TIER.team/enterprise` 和 `SELF_ENTERPRISE_FEATURE_FLAGS` 增加 `feat:audit`，路由用 `checkFeatureByPlan('feat:audit')`。同时补 API key feature-permission 映射 `feat:audit -> auditLogs:`，位置 `packages/server/src/services/apikey/index.ts:49-57`。A4 将菜单/路由切换后移除 `feat:login-activity`；不能同时把两者当长期产品 flag。

## 8. A0 结论与风险

-   可复用：self IAM actor 数据、权限中间件、tier feature gate、四库 full/ship migration 模式、现有登录活动 UI 的日期筛选/分页/i18n 结构。
-   必须新写：`FlowOpsAuditLog`、fail-safe audit service、数据库级筛选分页/导出、全部 admin/auth 挂点、组织作用域、`auditLogs:view`、`feat:audit` 和新 UI。
-   角色补权采用版本化四库迁移；不做启动时隐式对账。
-   login-activity 最终统一进新表；A2 起只写一份，旧表先保留历史/回滚数据，不再读写。
-   `feat:audit` 归 team/enterprise；free/pro 关闭，owner/admin 有 `auditLogs:view`，member 无。
-   两项后续实现风险必须在 A1/A2 明确测试：审计失败绝不能回滚主业务；查询必须强制组织隔离，不能复刻当前全库登录活动查询。
-   本轮未修改任何运行代码；等待 A0 人工评审后再进入 A1。

### 9.1 A5 真机矩阵

-   在 `FLOWOPS_IAM=self`、本机 commercial 满血、全新 SQLite 库上，通过真实 HTTP/cookie 会话完成角色、工作区和成员的建/改/删，以及邀请、注册、登录失败、登录、登出、组织用户更新/删除。调用方组织共得到 22 条审计记录，覆盖 16 个 action；admin 动作的 actor、target、before/after、status、IP 和 User-Agent 均与请求及数据库实体一致。
-   `role.*` 返回 3 条角色事件，`auth.*` 返回 9 条认证事件，`role.create` 精确过滤只返回一条；actor、时间范围和分页过滤通过。CSV 导出把以 `=` 开头的角色名称加前缀转义。
-   member 调用 `/api/v1/audit` 返回 403；free 模式下 `feat:audit=false`、API 返回 403“该功能需专业版”，全新浏览器侧栏不显示审计入口且直接访问 `/audit` 进入未授权页；本机 commercial/enterprise 模式下入口与页面可见。
-   第二组织 owner 即使传入第一组织的 `organizationId`，也只能查询第二组织记录；第一组织查询同样不包含第二组织数据，证明 query 参数不能覆盖认证组织作用域。
-   旧 `/login-activity` 浏览器路由跳转到 `/audit?action=auth.*`，页面实际展示 `auth.login/auth.logout/auth.loginFailed/auth.register`；兼容接口读取新表，返回 9 条，与同组织 `auth.*` 查询计数一致。

### 9.2 升级 fixture

-   PostgreSQL 真实 fixture 使用 `flowops_user.id uuid` 与 `flowops_login_activity.userId text`：4 条旧记录迁移后得到 4 条审计记录，code 映射保持不变；已知用户回填邮箱，未知用户和空 userId 的邮箱为 null，旧表保留。
-   老库角色 fixture 初始 owner/admin/member 均无 `auditLogs:view`。执行 `GrantAuditLogsPermission1779200000000` 后 owner/admin 各补一个 key、member 不变；重复执行结果完全一致，验证幂等。
-   `flowops_login_activity` 继续作为回滚兼容物理表保留，但运行时不再读写；是否物理删除留给后续独立清理迁移，不在本计划内隐式丢弃历史数据。
