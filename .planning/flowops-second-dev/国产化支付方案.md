# FlowOps 国产化支付与授权方案

> 状态：设计方案（v1）。基于现有 `codex/commercialization-v1` 分支的支付/计费骨架，结合 FlowOps「私有部署优先 · 信创/合规」定位编写。
> 本文是**设计方案**，不是执行清单；确认方向后可再拆成 Codex 执行计划或我直接落 P0。

---

## 0. 一句话结论（必须先想清楚的事）

对"私有部署优先"的 FlowOps，**「国产化支付」不等于「接个支付宝」**。它本质是**两套不同的钱**：

1. **在线收款**（支付宝/微信/银联/数字人民币）—— 服务**云版 / 个人版 / 小团队**，按订阅或用量；
2. **对公采购 + 离线授权（License）** —— 才是**企业 / 信创 / 政企私有部署**的**主收入路径**。这类客户的服务器**根本连不上你**（内网/气隙/涉密），**不能靠运行时联网计费**，只能靠**离线验签的授权文件**。

> ⚠️ 架构红线：**收款系统跑在你的公网商业化后台，被私有部署的产品端只做离线 License 校验，绝不回调你的服务器。** Codex 现在把支付逻辑放在 `packages/server` 里——这适合云版，但私有部署版必须把"收款+发证"剥离出去。

## 2. 核心框架：三层客户 × 收款模型

| 维度 | 个人 / 小团队（云·在线） | 企业私有部署（对公·License）**主力** | 政企 / 信创 / 涉密（气隙·线下） |
|---|---|---|---|
| 付什么 | 月/年订阅、用量包 | 年度授权 + 维保 / 永久授权 | 同左，强审计 |
| 怎么付 | 支付宝/微信/银联/数币 **在线扫码** | **对公转账**（询价→合同→回款核销）；可选在线 | **纯线下对公**，可能走集采/招投标 |
| 怎么交付 | 在线开通订阅（`BillingSubscription`） | **签发 License 文件**导入 | **License 文件 U 盘/介质**导入 |
| 是否联网 | 是 | **否**（产品端不联网） | **绝对否**（气隙） |
| Codex 骨架适配 | ✅ 直接服务 | ⚠️ 需补 License + 对公 | ⚠️ 需离线发证流程 |

> 结论：在线支付骨架先服务"个人/云"这层把**收款链路**跑通；**企业/信创主力**靠 **License 体系**，这才是要补的大头。

## 3. 国产收款渠道选型

### 3.1 直连 vs 聚合
- **直连**（支付宝开放平台 + 微信商户平台 + 银联）：无抽佣、数据自控、**信创友好**、合规清晰。**推荐主路径**。
- **聚合**（易宝/连连/Ping++/BeeCloud 等）：一次对接多渠道、起步快，但有费率、依赖第三方 SaaS。**仅建议作为云版快速起步**；私有部署/信创**慎用**（外部依赖与数据出域问题）。

### 3.2 渠道清单与接入方式
| 渠道 | 主用接口 | 适用场景 |
|---|---|---|
| 支付宝 | 当面付（扫码）、电脑网站支付、手机网站支付 | 个人/云在线 |
| 微信支付 | Native（扫码）、JSAPI、H5、APP | 个人/云在线 |
| 银联 / 云闪付 | 网关支付、云闪付 | 政企偏好、补全 |
| **数字人民币 e-CNY** | 经运营机构（工/建/中行等）受理，对公/对私 | 政企/信创加分项 |
| **对公转账** | 线下汇款 + **后台回款核销** | 企业/政企**主力** |

### 3.3 资质（红线）
- 收款主体 = **公司**（营业执照）→ 申请**企业支付宝/微信商户号**（**一清**，直连结算到你公司账户）。
- **严禁二清**：替他人代收/分账给第三方而无支付牌照 = 违规。FlowOps 是为自己产品收款（一清），合规。
- 沙箱→生产需**真实商户资质 + 审核周期**，提前申请。

## 4. 离线授权（License）体系 —— 私有部署的命脉

### 4.1 为什么必须做
气隙/内网/信创客户的 FlowOps **连不上你的服务器**，无法运行时校验订阅/用量。唯一可行：**离线签名授权文件**。

### 4.2 License 设计
- **格式**：签名的授权文件（JWT 风格或自定义），算法 **Ed25519 或 RSA-2048**。
- **载荷**：`{ 客户主体, 版本/edition, 启用模块[], 席位/并发上限, 签发时间, 到期时间, 机器指纹[], license_id, 备注 }`。
- **验签**：产品端内置**公钥**校验；**私钥只存在于你的发证后台**（KMS/HSM/隔离机）。
- **机器指纹**：主板 UUID / CPU / 网卡 MAC / 部署 ID 组合哈希，防止授权随意搬迁；支持多节点/集群授权（指纹白名单）。
- **到期与宽限期**：到期后**转只读或限频**，并留**宽限期**（如 15 天），**绝不业务硬中断**——政企最忌"突然不能用"。
- **吊销/续期**：在线版可下发 CRL；离线版靠到期 + 重新签发新文件。

### 4.3 发证流程
```
采购付款成功
  ├─ 对公：银行到账 → 后台人工/自动核销
  └─ 在线：支付回调验签成功
      ↓
发证后台用私钥签发 License 文件
      ↓
交付（下载 / 邮件 / U盘介质）
      ↓
客户在 FlowOps「授权管理」页导入 → 本地公钥验签 → 放开功能/席位
```

### 4.4 防破解的务实态度
技术只做"**防君子**"（提高破解成本）；真正的护城河是**合同 + 法务 + 审计日志**。不追求绝对防护，避免过度工程伤害正版体验。

## 5. 架构边界（最关键的一张图）

```
┌─────────────────────── 你的公网商业化后台（你运营）───────────────────────┐
│  官网/控制台  ·  收款(支付宝/微信/银联/数币)  ·  对公回款核销             │
│  发证服务(私钥签 License)  ·  开票  ·  合同/报价  ·  对账/退款            │
└───────────────────────────────┬──────────────────────────────────────────┘
                                 │  仅"交付 License 文件"（一次性、离线）
                                 ▼
┌──────────────── 客户私有部署的 FlowOps（信创/内网/气隙）──────────────────┐
│  只做：离线公钥验签  ·  按 License 放开功能/席位  ·  本地审计日志          │
│  ❌ 绝不回调你的服务器、❌ 不内置任何收款逻辑                              │
└───────────────────────────────────────────────────────────────────────────┘
```

> Codex 当前把 `controllers/payment` + `services/payment` 放进 `packages/server`：
> - **云版**：可以保留在产品里（云版本来就联网）。
> - **私有部署版**：应将"收款 + 发证"独立为**单独服务/独立仓**，产品内只保留 `LicenseVerifier`（公钥验签器）。用构建开关/包拆分隔离，避免把支付密钥与逻辑打进客户机。

---

## 6. 发票 / 合同 / 对账（企业必备，常被忽略）

- **发票**：增值税**电子普票/专票**。对接第三方电子发票（诺诺网 / 百望 / 航天信息）或初期人工开票；新增 `Invoice` 实体 + 申请流。专票是政企硬需求。
- **合同/报价**：报价单 → 合同 → 对公 → **回款核销**。可先"线下签 + 后台登记"，后续接电子签（e签宝/法大大）。
- **对账（reconciliation）**：
  - **主动查单补偿**：回调可能丢，必须定时查支付方订单状态兜底；
  - **日切对账文件**：每日拉支付宝/微信对账单与本地订单核对；
  - **幂等**：`orderNo` 唯一 + 回调去重（防重复发货/重复加额度）。
- **退款**：原路退款接口 + 审批流。

## 7. 合规与资质 Checklist

- [ ] 收款主体为公司，**企业商户号（一清）**，不做二清；
- [ ] 资金/数据合规、（必要时）等保备案；
- [ ] 价格公示、电子合同/电子签；
- [ ] 数字人民币需经**运营机构受理**；
- [ ] 私钥/商户密钥 **KMS 隔离**，不进代码库、不进客户机；
- [ ] 沙箱→生产前完成商户资质申请与审核。

## 8. 在现有代码上的增量（具体落点）

**Provider 扩展（实现现有 `PaymentProvider` 接口，零改动底座）**
- `UnionPayProvider`、`DCEPProvider`（数字人民币）、`OfflineTransferProvider`（对公·后台核销）。

**新增实体**
- `Invoice`（发票）、`License`（授权）、可选 `Contract`/`Quote`。

**新增服务**
- `LicenseService`：`issue / verify / revoke`（Ed25519 + 机器指纹 + 到期宽限）；
- `ReconciliationJob`：主动查单 + 日切对账；
- `RefundService`：退款 + 审批。

**订单状态机**
- `created → paid → delivered(License签发/订阅开通) → invoiced`；旁路 `refunded / closed / failed`。

**安全要点**
- 验签：支付宝 RSA2、微信 v3（AES-GCM + 平台证书轮换）；
- 金额一律以**分**为单位（`amountCents` 已是，保持）；
- 回调幂等 + 验签前置；密钥环境隔离。

**分层**
- 用独立包/构建开关把"收款+发证"与"产品端 License 校验"拆开（见 §5）。

## 10. 风险与强建议

1. **不要把收款逻辑塞进被私有部署的产品里**——数据、可用性、密钥安全全错位（§5 红线）。
2. **二清红线**：只为自己产品一清收款。
3. **License 防破解务实**：技术防君子，护城河在合同/法务/审计。
4. **资质前置**：商户号/发票/数币受理都有审核周期，早申请。
5. **到期不硬断**：留宽限期与只读降级，保政企口碑。
