# FlowOps 权限与付费改造计划（扣子/Dify 化）

> 目标：把 Flowise 那套"对用户不友好"的企业级权限 + 企业计费，改造成**扣子(Coze)/Dify 那种"少量固定角色 + 套餐/资源点"的友好模型**。
> 说明：撰写时 Web 检索工具不可用，扣子/Dify 的**模型结构**来自既有认知（准确），但**具体价格数字**可能有出入，落地前需复核官网。

---

## 0. 核心策略（先读这一条）

**不重写 Flowise 的鉴权引擎**——那是 fork 里最危险、最自杀的改动（鉴权安全攸关，且全代码耦合）。
正确做法是 **"在现有引擎上套一层简化"**：

- **权限**：锁定 3–4 个**预置角色** + 折叠"组织"层 + 隐藏颗粒权限编辑 → 体感等于扣子的"空间 + 成员"。引擎不动。
- **付费**：建一个统一 **"权益(Entitlement)层"**（套餐 + 资源点 + 功能开关），云版接在线订阅、私有版接 License/本地激活。

一句话：**底层照旧，体验换成扣子那种。** 把核心文件改动降到最低。

---

## 1. 现状：Flowise 的权限/付费为什么"不友好"（基于代码）

### 1.1 权限（重）
- `enterprise/rbac/Permissions.ts`：**~70 个颗粒权限 × 17 类**（chatflows×9、agentflows×9、documentStores×8、workspace×8、credentials×5、templates×6……），每个权限还分 `isOpenSource/isEnterprise/isCloud`。
- 两级层级：`organization` → `workspace`，加 `role`(可自定义) + `organization-user`/`workspace-user` 绑定 + `login-method`/`login-session` + SSO + 审计。
- `enterprise/rbac/PermissionCheck.ts` 在每个接口做校验。
- **痛点**：个人/中小团队要先建组织、建空间、定角色、勾几十个权限才能用——过度设计。

### 1.2 付费
- Flowise 企业版/云版自带企业计费；你已把在线计费/支付整套删除（`refactor(commercial)` 提交），现在只剩 `FLOWOPS_LOCAL_COMMERCIAL=true` 这个**无校验的"君子协议"开关**。

## 2. 参考：扣子(Coze) / Dify 怎么做的（友好在哪）

| 维度 | Flowise（现状） | 扣子 Coze | Dify |
| --- | --- | --- | --- |
| 空间层级 | 组织 + 空间 两级 | 个人空间 / 团队空间（扁平） | 工作空间（扁平） |
| 角色 | **自定义 + ~70 颗粒权限** | **3 个固定**：所有者/管理员/成员 | **4 个固定**：Owner/Admin/Editor/Normal |
| 上手 | 建组织→空间→角色→勾权限 | 进去就能用，邀成员选角色 | 同 Coze |
| 付费 | 企业计费 | **资源点(credits) + 会员**(免费/专业)，可充值 | **套餐**(Sandbox/Pro/Team) + message credits |
| 计费粒度 | 席位/企业合同 | 按模型调用扣"资源点"，月度赠送+充值 | 按消息/credits，套餐含额度 |
| 哲学 | 颗粒可控、面向大企业 IT | **少量固定角色 + 套餐/额度，开箱即用** | 同 Coze |

**结论**：扣子/Dify 的"友好"= **固定少量角色** + **套餐/资源点额度**。我们照搬这套哲学。

## 3. 目标模型（FlowOps 版）

### 3.1 权限：锁 4 个预置角色 + 折叠组织层
对用户只暴露 **"空间(workspace)"** 一个概念（组织层自动建默认组织、UI 隐藏）。固定 4 角色，**不让用户编颗粒权限**：

| 角色 | 能力 | 映射 Flowise 权限集 |
| --- | --- | --- |
| **所有者 Owner** | 全部 + 计费/成员/删空间 | 全量权限 |
| **管理员 Admin** | 管内容 + 管成员，不能删空间/改计费 | 全量减 `workspace:delete`、计费 |
| **成员 Member** | 建/改/跑智能体与资源 | chatflows/agentflows/tools/... 的 view/create/update/delete/export/import/config |
| **访客 Viewer** | 只看 + 运行，不可改 | 各类 `:view` + 运行 |

- **实现**：角色 = 现有权限引擎里的一个权限 bundle。我们只**预置**这 4 个 bundle，引擎零改。
- **隐藏**：颗粒权限编辑 UI、自定义角色入口、组织管理 UI、（私有/团队不需要时）SSO 入口。
- 🔒 **安全红线**：简化只在 **UI + 预置** 层；底层 `PermissionCheck` 仍真实生效——**"隐藏权限编辑" 绝不等于 "绕过权限校验"**。

### 3.2 付费：统一"权益(Entitlement)层" + 资源点
建一个与"来源"解耦的权益抽象：

```
Entitlement = { tier, seats(席位), credits(资源点/月), creditBalance(可充值余额),
                features[], concurrency, expireAt }
```

**套餐分层（示例，价格待定）**：

| 套餐 | 席位 | 资源点/月 | 功能 |
| --- | --- | --- | --- |
| 免费版 | 1–3 | 少量赠送 | 基础节点、单空间 |
| 专业版 | 更多 | 较多 + 可充值 | 高级节点、并发、去水印 |
| 团队版 | 多 | 多 | 多空间、成员管理 |
| 企业/私有版 | 自定义 | 自定义 | 私有部署 License、（可选）SSO/审计 |

- **资源点(credits)**：按模型调用/节点消耗扣点，月度赠送 + 可充值（扣子式）。**复用**刚被删 billing 里的计量点（`utils/quotaUsage.ts`、`buildChatflow.ts` 的埋点是现成的）。
- **双来源（关键设计）**——同一套"权益校验中间件"消费 Entitlement，不关心来源：
  - **私有部署版**：权益来自 **License 文件**（tier/seats/到期/机器指纹）或 `FLOWOPS_LOCAL_COMMERCIAL` → 映射到某 tier。
  - **云/SaaS 版**（若做）：权益来自**在线订阅 + 资源点充值**（接国产支付）。

## 4. 实施路线（分阶段、低分叉、可回退）

| 阶段 | 目标 | 内容 | 风险 |
| --- | --- | --- | --- |
| **P0 权限简化** | 体感立刻"像扣子" | 预置 4 角色、默认组织自动建、隐藏颗粒权限/自定义角色/组织 UI。**引擎零改，纯预置+前端** | 低 |
| **P1 权益层** | 重建友好计费内核 | `Entitlement` 抽象 + 资源点/quota 计量（复用 quotaUsage 埋点）+ 功能开关；先接"License/本地激活"来源（私有版）；顺带把"君子协议"升级成**签名 License** | 中 |
| **P2 资源点/套餐 UI** | 扣子式付费页 | 套餐页、资源点余额/用量看板、升级引导、到期宽限提示 | 中 |
| **P3（可选）云版在线支付** | 仅做云/SaaS 才需要 | 接国产支付（支付宝/微信）给在线订阅充资源点；本地版**不需要** | 高 |

> 注：私有部署优先的话，**P0+P1+P2 就够了**（License 驱动权益）；P3 只在你要做云版时才上。

## 5. Fork 纪律与风险（必须正视）

- ⚠️ **这是 fork 里最危险的区域**：鉴权安全攸关、全代码耦合（每个 controller 走 `PermissionCheck`，`IdentityManager` 无处不在）。
- 所以策略是 **"套层不重写"**：改动集中在
  - 角色**预置数据**（seed 默认 4 角色的权限 bundle）
  - **前端 UI**（隐藏颗粒/自定义/组织入口，换成扣子式空间+成员页）
  - **权益中间件**（新增，独立模块）
  尽量**不碰** `Permissions.ts`/`PermissionCheck.ts`/鉴权实体。每个核心文件改动进 `FORK-CHANGES.md`。
- **可回退**：P0 是 UI+预置，回退只需恢复 UI；不破坏数据。
- **别重复造轮子**：资源点计量直接复用现有 `quotaUsage` 埋点，不要新搭一套并行计量。

### B0. 角色层级：平台级 vs 空间级（4 + 1）

FlowOps 角色分两层，互不越界：

| 层级 | 角色 | 作用范围 | 专属能力 |
| --- | --- | --- | --- |
| **平台级** | 平台超级管理员（运营 / 部署管理员） | 整个部署 · 所有空间 | **看所有用户**、禁用/删号、管所有空间、管 License/计费、SSO/审计、系统设置 |
| **空间级** | 所有者 / 管理员 / 成员 / 访客 | 单个空间内 | 仅在本空间生效——看不到别的空间，更看不到全部用户 |

> - **没有任何空间角色能看「所有用户」**；这是平台超管的专属能力。
> - 私有部署：平台超管 = 持 License 的部署方（通常首个安装账号）；云版 = 平台运营。
> - 映射 Flowise：平台超管 ≈ 默认组织 owner + `users:manage` / `logs:view` / `loginActivity:view`；空间 4 角色 = workspace 内预置权限 bundle。
